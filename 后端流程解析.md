# 后端流程解析

这份文档专门讲后端。

如果你把整个项目想成一家工厂，前端像“接待大厅”，Agent 像“调度中心”，那后端就是“真正干活的生产车间”。文件上传、数据库读写、报告解析、知识检索、模型调用、小结导出，全部发生在这里。

---

## 1. 后端技术栈

后端使用的是：

- FastAPI
- SQLModel
- SQLite
- OpenAI 兼容调用方式接千问模型
- `pdfplumber` 处理文本型 PDF
- `reportlab` 生成 PDF

你可以这样理解这些技术在项目里的位置：

- FastAPI：负责对外提供 HTTP 接口
- SQLModel + SQLite：负责数据存储
- LLM：负责理解和生成
- `pdfplumber`：负责读报告里的文字
- `reportlab`：负责输出健康小结 PDF

---

## 2. 后端代码从哪里开始运行

后端入口在：

- `backend/app/main.py`

这个文件做了 4 件事：

1. 创建 FastAPI 应用
2. 配置跨域
3. 注册 API 路由
4. 在启动时初始化数据库和本地知识库

也就是说，当你运行：

```powershell
uvicorn app.main:app --reload --port 8000
```

真正启动的就是这里。

---

## 3. 启动时到底发生了什么

`main.py` 里最重要的是 `startup`。

后端启动后会按这个顺序做：

1. 调 `init_db()`
2. 确保数据库表存在
3. 打开一个数据库会话
4. 调 `knowledge_service.ensure_initialized(session)`
5. 如果知识库为空，就把本地知识种子写进数据库

这一步很关键，因为它意味着：

- 项目第一次启动时就会自动准备本地知识库
- 前端不需要再手动点一个“初始化知识库”按钮

---

## 4. 配置从哪里来

配置文件在：

- `backend/app/core/config.py`

这里定义了项目运行需要的核心参数，比如：

- API Key
- 模型名称
- WHO 凭据
- 短期上下文轮数
- 数据目录位置

这个文件的作用是“把环境变量整理成代码里好用的配置对象”。

你可以把它理解成：

“后端所有服务共享的一份总配置表。”

---

## 5. 数据库层是怎么组织的

数据库相关代码主要有两部分：

- `backend/app/core/database.py`
- `backend/app/models/entities.py`

### `database.py`

负责：

- 创建 SQLite 引擎
- 创建 Session
- 初始化表
- 给 FastAPI 的依赖注入提供 `get_session()`

### `entities.py`

负责定义表结构。这个项目里比较重要的表包括：

- `Report`
  - 保存一份报告的元信息和原始文本
- `ReportItem`
  - 保存一份报告里的每个结构化指标
- `ChatSession`
  - 保存会话
- `ChatMessage`
  - 保存消息记录
- `KnowledgeDocument`
  - 保存本地知识库内容
- `SummaryArtifact`
  - 保存生成的小结

小白可以先不用纠结 ORM 细节，只要知道：

- `entities.py` 决定“数据库里有什么表”
- `database.py` 决定“怎么连上数据库”

---

## 6. 后端 API 有哪些

路由都在：

- `backend/app/api/routes.py`

这是理解后端最重要的文件之一，因为它告诉你：

- 前端调用了哪些接口
- 每个接口最终落到哪个服务

当前项目主要接口如下。

### 6.1 上传报告

`POST /api/reports/upload`

用途：

- 接收用户上传的 PDF / JPG / PNG

流程：

1. 保存文件
2. 创建报告记录
3. 立刻返回初始报告对象
4. 把真正的解析工作交给后台任务

这意味着上传接口不会一直卡到解析结束。

### 6.2 查询报告

`GET /api/reports/{report_id}`

用途：

- 获取一份报告当前的解析结果

### 6.3 订阅报告解析进度

`GET /api/reports/{report_id}/stream`

用途：

- 用 SSE 推送解析进度

返回内容包括：

- 当前阶段
- 当前进度百分比
- 是否完成
- 最终解析结果

### 6.4 普通问答接口

`POST /api/agent/chat`

用途：

- 一次性拿到完整回答

### 6.5 流式问答接口

`POST /api/agent/chat/stream`

用途：

- 持续推送回答进度和正文片段

### 6.6 生成健康小结

`POST /api/summaries/generate`

用途：

- 根据报告生成 Markdown 和 PDF 小结

### 6.7 下载 PDF

`GET /api/summaries/{summary_id}/pdf`

用途：

- 下载生成好的 PDF

### 6.8 查看知识库来源

`GET /api/knowledge/sources`

用途：

- 查看当前知识库有多少条内容、按信任等级怎么分布

虽然当前前端主界面已经不再单独展示这块，但接口还保留着，方便调试和扩展。

## 7. 报告解析是怎么做的

报告解析核心在：

- `backend/app/services/report_service.py`

这是整个后端最像“业务主流程”的服务。

### 7.1 上传阶段

上传时先执行 `create_upload(...)`。

它做的事情包括：

1. 校验文件类型
2. 生成报告 ID
3. 把文件存到 `data/uploads/`
4. 在数据库里创建 `Report`
5. 初始状态设为 `uploaded`

### 7.2 后台解析阶段

真正解析由 `process_report(report_id)` 完成。

它大致分成这些步骤：

1. 更新进度为 `processing`
2. 读取文件路径
3. 判断文件类型
4. 尝试提取文本
5. 把文本转成结构化指标
6. 做异常判断
7. 保存结果
8. 更新最终状态

---

## 8. 文本提取是怎么做的

文本提取不是一刀切，而是分情况处理。

### 情况 A：文本型 PDF

优先走 `pdfplumber`。

优点：

- 快
- 准
- 不需要视觉模型

### 情况 B：图片或扫描件

如果配置了视觉模型，就走 OCR。

对应模型是：

- `QWEN_VL_MODEL`

也就是说：

- 普通 PDF 先走本地文本提取
- 扫描件才更依赖视觉模型

---

## 9. 结构化指标是怎么来的

文本拿到以后，并不是直接存一大段原文，而是要提取成结构化数据。

典型的结构化字段包括：

- 指标名
- 原始值
- 数值型值
- 单位
- 参考范围
- 状态
- 备注

这一部分使用：

- `llm_service.chat_json_fast(...)`

也就是说，后端会让 `fast` 模型按照固定 JSON 结构输出。

如果模型失败，代码还会退回到更保守的正则提取逻辑。

这里的设计思想是：

- 尽量用模型提高适配性
- 但不能完全依赖模型

---

## 10. 指标高低异常是谁判断的

这一点非常重要：

`指标高 / 低 / 正常` 的判定，不是大模型决定的。

而是后端规则判断。

比如：

- `3.9-6.1`
- `< 5.7`
- `>= 40`

后端会解析这些参考范围，再把指标值和范围进行比较，得到：

- `high`
- `low`
- `normal`
- `unknown`

这样做的好处是：

- 规则稳定
- 可解释
- 不会因为模型发挥不同而出现漂移

---

## 11. 报告进度为什么能流式显示

报告进度相关代码在：

- `backend/app/services/report_progress_service.py`

这个服务本质上是在内存里维护一份“当前解析状态”。

`report_service.process_report(...)` 每走到一个阶段，就会更新：

- 阶段名
- 标签
- 百分比
- 是否完成
- 是否失败

然后 `/api/reports/{id}/stream` 把这些状态通过 SSE 推给前端。

所以你看到前端不断变化的：

- 等待解析
- 提取文本
- OCR
- 结构化提取
- 保存结果
- 完成

其实就是后端在逐步推送这些状态。

---

## 12. 本地知识库怎么工作的

知识库相关代码在：

- `backend/app/services/knowledge_service.py`
- `backend/app/services/knowledge_seed.py`

### `knowledge_seed.py`

负责提供本地种子数据。

里面放的是：

- 常见体检指标
- 常见医学术语
- 常见检查发现

### `knowledge_service.py`

负责：

- 初始化种子数据
- 检索知识
- 按指标名匹配知识条目
- 统计知识来源

这个项目没有做联网爬虫知识库，而是采用：

`本地种子 + 检索 + 模型润色`

这样做的优点是稳定、简单、可控。

---

## 13. WHO 服务在后端里是什么角色

WHO 相关代码在：

- `backend/app/services/who_service.py`

它不是整个系统的通用知识底座，而是一个“增强型工具”。

当前主要用于：

- 医学名词解释
- 术语标准化
- ICD-11 结果增强

比如用户问：

- “甲亢是什么”

系统会尝试把它标准化成更容易命中的术语，例如：

- `甲状腺功能亢进`
- `甲状腺功能亢进症`

然后再去查 WHO。

WHO 结果如果命中，会和本地知识一起交给模型润色，并在回答末尾追加：

- `来源于 WHO ICD-11`

---

## 14. 模型调用层是怎么封装的

模型调用代码在：

- `backend/app/services/llm.py`

这个文件的作用是把模型调用统一封装起来，避免业务代码里到处写 API 细节。

它大致提供几类能力：

- 文本模型调用
- JSON 输出调用
- 流式文本输出
- 视觉模型 OCR

同时它还把模型分成了不同档位：

- `fast`
- `max`
- `vl`

这让业务层只需要关心：

- 这次任务是“快优先”还是“质量优先”

而不需要重复关心底层 API 细节。

---

## 15. 健康小结是怎么生成的

小结相关代码在：

- `backend/app/services/summary_service.py`

流程如下：

1. 读取报告
2. 检查报告是否解析完成
3. 提取异常项
4. 获取这些异常项的知识解释
5. 生成 Markdown 内容
6. 写出 PDF
7. 把结果保存到数据库

Markdown 内容优先尝试让 `max` 模型生成。

如果模型失败，则使用本地 fallback 文本拼装。

PDF 使用 `reportlab` 直接按行绘制。

这意味着这个项目的小结导出链，实际上也是“模型优先，模板兜底”的思路。

---

## 16. 一个请求进入后端后，通常会经过哪些层

这部分是理解后端最关键的抽象。

你可以把一次请求看成会经过下面几层：

### 第 1 层：路由层

在 `routes.py`

作用：

- 接收 HTTP 请求
- 解析参数
- 注入数据库会话
- 调用服务
- 返回响应

### 第 2 层：服务层

在 `services/`

作用：

- 处理业务逻辑
- 调模型
- 调知识库
- 做规则判断

### 第 3 层：存储层

在 `database.py` + `entities.py`

作用：

- 持久化报告、消息、知识、小结

所以如果你要读代码，建议不要从数据库类定义开始读，而是：

先看路由怎么进，再看服务怎么干活，最后回头看数据怎么存。

## 17. 小白应该先看哪些后端文件

建议顺序如下：

1. `backend/app/main.py`
2. `backend/app/api/routes.py`
3. `backend/app/services/report_service.py`
4. `backend/app/services/react_agent.py`
5. `backend/app/services/knowledge_service.py`
6. `backend/app/services/who_service.py`
7. `backend/app/services/summary_service.py`
8. `backend/app/core/schemas.py`
9. `backend/app/models/entities.py`

这个顺序的好处是：

- 先看主流程
- 再看复杂细节
- 不容易一开始就被表结构和模型包装细节淹没

---

## 18. 当前后端最重要的工程思想

如果把后端设计思路压缩成几句话，就是：

- 确定性的事情尽量不用模型
- 模型负责理解和表达，不负责所有真值判断
- 本地知识库优先，WHO 用于增强，模型用于润色
- 前端体验通过 SSE 改善
- 报告处理用异步背景任务，而不是同步卡死请求

这几条，是你理解这个后端最值得记住的东西。
