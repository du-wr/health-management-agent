# 项目流程解析

这份文档是整个 Demo 的总入口，目的是帮助第一次接触这个项目的人，先建立一张完整的“地图”。

如果你把这个项目想成一个真实产品，它做的事情其实很简单：

1. 接收用户上传的体检报告
2. 把报告解析成结构化指标
3. 围绕报告和医学问题做解释
4. 给出一份面向普通用户的健康小结

但是实际代码里，这 4 件事被拆成了前端、后端、Agent、多模型、知识库、流式输出等很多模块。小白最容易卡住的地方，就是“文件很多，但不知道先看哪里”。这份文档就是为了解决这个问题。

---

## 1. 这个项目到底是什么

这是一个中文医疗健康场景的 Agent Demo，主题是：

`体检报告解读与健康咨询`

它不是一个“万能医疗助手”，而是一个有明确边界的演示系统：

- 它能读常见体检报告
- 它能解释指标异常大概代表什么
- 它能回答部分医学名词是什么意思
- 它能生成一份面向普通用户的小结
- 它不会给出正式诊断
- 它不会给出处方和剂量

所以这个 Demo 的核心价值，不是“像医生一样下结论”，而是演示：

- 如何把上传、解析、问答、小结串成一个完整产品
- 如何把规则、安全、知识库和大模型混合起来使用
- 如何让前端通过流式体验看起来像一个真正的 Agent 产品

---

## 2. 一句话看懂整体架构

你可以把整个项目拆成 3 个层次：

### 第一层：前端

负责界面和交互：

- 上传报告
- 展示报告解析进度
- 与 Agent 对话
- 展示流式回答
- 生成和下载健康小结

### 第二层：后端

负责业务能力：

- 存文件
- 解析 PDF / 图片
- 保存数据到 SQLite
- 调用知识库、WHO、LLM
- 提供 HTTP API 和 SSE 流

### 第三层：主体 Agent

负责“怎么回答问题”：

- 先做医疗安全判断
- 再判断这是报告追问、名词解释还是其他问题
- 再决定是否查本地知识库、是否查 WHO、是否调用大模型
- 最后生成面向用户的答案

## 4. 一次完整使用流程是怎么走的

下面用最典型的使用方式来理解整个链路。

### 场景 A：用户上传一份体检报告

流程如下：

1. 用户在前端选择一个 PDF、PNG 或 JPG 文件
2. 前端把文件发给后端 `/api/reports/upload`
3. 后端先把文件存到磁盘，再创建数据库记录
4. 后端启动后台任务解析报告
5. 前端通过 SSE 持续订阅报告解析进度
6. 解析完成后，前端拿到结构化报告结果
7. 页面显示异常指标、原始文本、解析状态

### 场景 B：用户围绕报告继续提问

例如：

- “分析一下我的报告指标”
- “这些异常代表什么”
- “哪个最需要优先关注”

流程如下：

1. 前端把问题和 `report_id` 发给 Agent
2. Agent 先看这是不是高风险问题
3. 如果不是高风险，就做输入分析和问题改写
4. Agent 判断这是“报告追问”
5. Agent 读取这份报告的异常项
6. Agent 结合本地知识库解释单个指标
7. Agent 再做一次“综合异常整理”
8. 最后由 `max` 模型润色成对用户友好的答案
9. 前端通过流式方式一段段显示答案

### 场景 C：用户问一个医学名词

例如：

- “甲亢是什么”
- “灰指甲是什么病”

流程如下：

1. Agent 判断这是“名词解释”
2. 先查本地知识库
3. 如果配置了 WHO，就进一步查 WHO ICD-11
4. 把本地知识和 WHO 结果交给模型做润色
5. 输出一段更自然、更容易理解的解释

## 5. 为什么这个项目不是“纯大模型直接回答”

很多人第一次看会觉得：

“既然大模型可以直接回答，那为什么还要做这么多层？”

原因是医疗场景对稳定性要求更高。

如果完全让模型自由生成，会有几个明显问题：

- 容易幻觉
- 容易越过医疗边界
- 同一个问题前后回答不稳定
- 难以把报告里的真实指标和回答绑定起来

所以这个 Demo 采用的是混合架构：

- 安全边界：规则
- 报告结构化：规则 + 模型
- 知识来源：本地知识库 + WHO
- 最终解释：模型

这种方式的思路是：

“把确定性的事情交给规则和知识库，把需要表达能力和综合能力的事情交给大模型。”

## 6. 多模型在这个项目里是怎么分工的

这个项目不是只接了一个模型，而是做了分工。

### `fast` 模型

负责轻量、快节奏的任务：

- 输入分析
- 问题改写
- 报告问题拆解
- 批量指标解释的结构化整理

### `max` 模型

负责高质量输出：

- 最终回答润色
- 更自然的中文表达
- 健康小结生成

### `vl` 模型

负责视觉相关任务：

- 图片 OCR
- 图片类报告文本提取

这套分工的目的很明确：

- 前置逻辑尽量快
- 最终结果尽量好
- 不让最贵、最慢的模型承担所有步骤

## 7. 小白建议按什么顺序读代码

如果你是第一次看这类项目，建议按这个顺序：

### 第一步：先看总入口

- `backend/app/main.py`
- `frontend/src/main.tsx`
- `frontend/src/App.tsx`

先建立“程序从哪里启动”的概念。

### 第二步：看 API 路由

- `backend/app/api/routes.py`

这一步的目标是搞清楚：

- 前端会请求哪些接口
- 每个接口最后调了哪个服务

### 第三步：看报告服务

- `backend/app/services/report_service.py`

这是整个 Demo 最像“业务系统”的一段代码，因为它处理文件上传、文本提取、结构化解析、异常判断。

### 第四步：看 Agent 核心

- `backend/app/services/react_agent.py`
- `backend/app/services/prompt_templates.py`

这是整个项目最重要、也最复杂的部分。

### 第五步：看知识库与 WHO

- `backend/app/services/knowledge_service.py`
- `backend/app/services/knowledge_seed.py`
- `backend/app/services/who_service.py`

### 第六步：再回头看前端

- `frontend/src/App.tsx`
- `frontend/src/api.ts`
- `frontend/src/types.ts`

这时候你会更容易理解前端为什么这样组织状态和流。

## 9. 当前版本有哪些重要取舍

这个 Demo 已经能跑通，但它不是生产级系统。当前有几个很明确的工程取舍：

### 使用 SQLite

优点是简单、轻量、适合演示。

代价是：

- 并发能力有限
- 不适合复杂部署场景

### 使用 `BackgroundTasks`

优点是实现简单，演示方便。

代价是：

- 更适合单机 Demo
- 不适合高并发任务队列

### 使用本地知识种子

优点是稳定、可控。

代价是：

- 覆盖面一定有边界
- 需要持续补充内容

### WHO 只接在术语解释链

优点是职责清晰。

代价是：

- 它目前不是整个系统的“通用知识后端”
