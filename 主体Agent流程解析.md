# 主体 Agent 流程解析

这份文档专门讲整个 Demo 最核心的部分：

`Agent 到底是怎么思考和回答问题的。`

如果你前面已经看过总览、后端和前端，这份文档就是最后一块拼图。

---

## 1. 先说结论：这个 Agent 不是“纯模型自由发挥”

这个项目里的 Agent 采用的是混合式设计：

- 安全边界用规则
- 路由和拆解用 `fast` 模型
- 最终表达用 `max` 模型
- 知识来源优先使用本地知识库和 WHO

所以它不是：

- 完全靠规则跑

也不是：

- 完全把所有事情都交给大模型

而是：

`把能确定的事情尽量结构化，把需要综合和表达的事情交给模型。`

这就是整个 Demo 的 Agent 思路。

---

## 2. Agent 代码主入口在哪里

最核心的文件是：

- `backend/app/services/react_agent.py`

提示词模板在：

- `backend/app/services/prompt_templates.py`

你可以把这两个文件理解成：

- `react_agent.py` 负责流程编排
- `prompt_templates.py` 负责告诉模型“应该怎么做”

---

## 3. 为什么它叫“自定义 ReAct”

经典 ReAct 的思路是：

- 先思考
- 再行动
- 再观察
- 再继续思考

这个项目虽然没有完全照搬论文里的自由形式 ReAct，但保留了类似骨架：

1. 理解问题
2. 选择工具
3. 获取观察结果
4. 生成最终回答

只是它做了比较强的约束：

- 不允许模型无限制地自由调用工具
- 安全层放在最前面
- 很多步骤先结构化再进入最终生成

所以可以叫：

`受约束的、自定义 ReAct`

---

## 4. Agent 的最顶层执行顺序

不管是同步回答还是流式回答，整体思路都差不多。

一次用户问题进入后，通常会按这个顺序走：

1. 安全检查
2. 获取短期上下文
3. 输入分析
4. 按意图走不同分支
5. 调用知识库 / WHO / 报告工具
6. 组织答案
7. 完整性校验
8. 返回结果

理解了这 8 步，整个 Agent 就不再神秘。

---

## 5. 第一步：安全检查

安全规则在：

- `backend/app/services/safety_service.py`

这一步的目的很明确：

- 先判断这是不是高风险问题

比如：

- 要求诊断
- 要求开药
- 要求给剂量
- 可能是急症

如果命中高风险，系统不会继续正常解释，而是直接进入：

- `safety_handoff`

也就是转成“建议及时就医”的保守回答。

这一步非常重要，因为医疗场景里，安全边界必须先于智能能力。

---

## 6. 第二步：短期上下文

Agent 不只是看用户当前这一句话，还会读取同一 `session_id` 最近几轮对话。

相关逻辑在：

- `react_agent.py` 中的最近上下文读取函数

这个能力主要是为了解决这些问题：

- “那这个呢？”
- “和前一个比哪个更重要？”
- “你刚才说的复查是指哪一项？”

也就是说，它不是完整长记忆，而是：

- 最近 N 轮短上下文

默认通过配置控制。

---

## 7. 第三步：输入分析

这是当前 Agent 非常关键的一层。

不是用户问什么，系统就直接照原话去查，而是会先让 `fast` 模型做一次输入分析。

输入分析的目标包括：

- 识别意图
- 改写问题
- 标准化术语
- 判断是否查本地知识库
- 判断是否查 WHO
- 判断是否需要 `max` 模型

这个步骤能解决很多现实问题，比如：

- 用户说的是俗称，不是标准医学术语
- 用户问题很短，但实际是报告追问
- 用户问题依赖上一轮上下文

例如：

- “甲亢是什么”

可能会被标准化成：

- `甲状腺功能亢进`

这样 WHO 更容易命中。

---

## 8. Agent 里最核心的几种意图

当前主要意图有这些：

### `report_follow_up`

围绕已上传报告继续追问。

例如：

- “这些异常是什么意思”
- “分析一下我的报告指标”

### `term_explanation`

解释一个医学名词。

例如：

- “甲亢是什么”
- “灰指甲是什么病”

### `symptom_rag_advice`

围绕症状做科普性建议。

### `collect_more_info`

当前信息不足，需要用户补充。

### `safety_handoff`

高风险问题，停止普通回答。

Agent 先判意图，再决定后面走哪条分支。

---

## 9. 报告追问分支是怎么工作的

这是当前系统最复杂、也最有代表性的分支。

当输入被判成 `report_follow_up` 后，不是直接把报告全塞给大模型，而是分层处理。

### 9.1 先读取报告

系统会拿到：

- 全部结构化指标
- 异常项
- 报告状态

如果报告还没解析完成，系统会先返回“请等待解析完成”的提示。

### 9.2 再做问题拆解

这里会让 `fast` 模型生成一个结构化计划。

这个计划通常包含：

- `focus_item_names`
- `need_item_explanations`
- `need_synthesis`
- `need_next_steps`
- `synthesis_axes`
- `follow_up_needed`

也就是说，Agent 会先回答这些问题：

- 这一轮该重点解释哪些指标
- 是否要逐项解释
- 是否要做综合分析
- 是否要给后续建议

这一步非常重要，因为它把一个宽泛问题拆成了清晰任务。

### 9.3 解释单个指标

然后 Agent 会调用本地知识库，对这些重点指标做逐项解释。

这里的解释通常包括：

- 指标的基本意义
- 异常方向代表什么
- 常见影响因素
- 需要关注什么

### 9.4 做综合异常整理

这是后来补上的一层，也是让回答更稳定的关键。

系统会把多个单项解释整理成一份综合素材，通常包括：

- `summary`
- `priority_axes`
- `combined_findings`
- `next_steps`

例如它会尝试总结：

- 这批异常主要集中在哪个风险方向
- 哪些指标可能互相关联
- 哪些是主问题，哪些只是背景线索
- 下一步应该怎么复查或就诊

### 9.5 生成最终回答

有了：

- 报告异常项
- 单项解释
- 综合异常素材

之后，再交给 `max` 模型进行润色。

注意这里的重点是：

`max` 不再从零开始胡写，而是基于已经准备好的结构化材料做表达优化。

这比早期“让模型直接生成整段报告解读”要稳定得多。

---

## 10. 为什么报告类问题之前容易只剩标题

这是这个项目开发过程中一个很典型的问题。

之前的链路里，最终回答过度依赖 `max` 模型自由发挥，导致模型有时只会输出：

- 主要异常解读
- 综合解读
- 后续建议

却不把内容展开。

后面修复的核心思路是：

1. 先做输入分析
2. 再做报告问题拆解
3. 再做单项解释
4. 再做综合异常整理
5. 最后才让 `max` 做润色

也就是说：

`先把内容准备好，再让模型组织语言。`

而不是：

`只给模型一个宽泛任务，让它自己临场编全文。`

---

## 11. 名词解释分支是怎么工作的

当输入被判成 `term_explanation` 后，流程如下：

### 11.1 先查本地知识库

如果本地知识库里有这个术语，就先拿到一批解释材料。

### 11.2 再尝试查 WHO ICD-11

如果配置了 WHO 凭据，就会进一步查询 WHO。

这里不是只拿用户原话硬查，而是会尝试多个候选词：

- 原问题
- 清洗后的术语
- 本地知识命中的标准标题
- 别名

这样像：

- “甲亢”

这种口语化词，也能更容易命中：

- `甲状腺功能亢进`

### 11.3 最后由模型润色输出

系统会把：

- 本地知识
- WHO 结果
- 用户问题

一起交给 `max` 模型生成面向普通用户的解释。

如果 WHO 真命中了，正文末尾还会追加：

- `来源于 WHO ICD-11`

---

## 12. 为什么名词解释不能完全靠大模型现编

如果只是普通聊天机器人，直接回答“甲亢是什么”当然也能给出看起来不错的答案。

但这个系统想和“裸大模型”拉开差异，就必须增加外部知识来源。

本地知识库和 WHO 的价值在于：

- 来源更稳定
- 可追溯
- 幻觉更少
- 更容易做边界控制

所以这里的思路不是：

“模型自己知道一切”

而是：

“模型基于受控知识源来表达”

---

## 13. `fast` 和 `max` 分别做什么

这是当前 Agent 设计里非常重要的一个点。

### `fast`

负责：

- 输入分析
- 问题改写
- 术语标准化
- 报告拆解
- 轻量结构化任务

### `max`

负责：

- 最终回答润色
- 更自然的中文表达
- 总结和收口

这套分工解决的是一个工程问题：

- 如果所有步骤都用最强模型，会太慢
- 如果所有步骤都用快模型，最终文字质量又不够

所以采用：

`前面快，后面强`

---

## 14. 流式输出是怎么和 Agent 结合的

Agent 不只是一次性返回完整字符串，它还支持流式输出。

当前流式接口会推送：

- 会话信息
- 状态信息
- 正文片段
- 最终结果

这里有一个重要设计：

后来为了避免“前面生成一版，最后又突然跳成另一版”的问题，流式链被改成：

- 先生成最终可接受答案
- 再按段落或片段流给前端

这样做的好处是：

- 前端看到的流式内容，就是最终答案的逐步展开
- 不会先看到草稿，最后又被覆盖成另一个版本

---

## 15. Agent 为什么还需要 fallback

即便现在已经有：

- 输入分析
- 问题拆解
- 综合异常层
- 本地知识库
- WHO
- 多模型分工

Agent 仍然需要 fallback。

原因很简单：

- 模型可能返回空结果
- 模型可能只输出提纲
- 模型可能格式不符合预期
- 外部依赖可能失败

所以系统里有一套“半成品检测”和“兜底输出”机制。

简单理解就是：

如果模型输出明显不完整，系统不能把半成品直接交给用户，而要修复或退回到本地结构化答案。

---

## 16. 为什么还要保留规则，而不是全交给 Agent

很多人第一次做 Agent 会想：

“既然有模型，为什么还要保留那么多规则？”

在这个项目里，保留规则有 3 个原因：

### 1. 医疗安全

有些边界不适合让模型自由决定。

### 2. 结构化任务更适合规则

比如参考范围比较、状态判定。

### 3. 工程稳定性

完全靠模型会更慢，也更不稳定。

所以这个项目的 Agent 不是“规则越少越先进”，而是：

`该规则的地方规则化，该模型的地方模型化。`

---

## 17. 调试信息为什么要暴露给前端

为了让这个 Demo 更适合学习，Agent 会把一些中间结果放到 `debug` 里返回：

- `analysis`
- `plan`
- `synthesis`
- `used_tools`

这样你在页面上就能看到：

- 输入分析是怎么判的
- 报告问题被拆成了什么
- 综合异常层长什么样

如果没有这一层，你看到的就只是“输入一句话 -> 输出一句话”，很难学到真正的 Agent 结构。

---

## 18. 小白如何阅读 Agent 代码

建议顺序如下：

1. 先看 `react_agent.py` 里最顶层的 `respond` 和 `stream_respond`
2. 再看输入分析相关函数
3. 再看 `report_follow_up` 分支
4. 再看 `term_explanation` 分支
5. 再看 fallback 和 repair
6. 最后再看 `prompt_templates.py`

为什么不建议先看 prompt？

因为如果你一开始就看一大堆 prompt，很容易只看到“提示词写了什么”，却不知道它在整个系统里的位置。

先理解流程，再看 prompt，效率更高。

---

## 19. 这个 Agent 当前最重要的优点

如果只总结几个最关键的优点，我会列这几个：

- 有明确医疗安全边界
- 报告问题不是简单关键词回复，而是有拆解和综合层
- 术语解释可以接 WHO，权威性更强
- 多模型分工清晰
- 支持流式输出和调试可视化

这几条让它不只是“能跑”，而是“能讲得清楚自己怎么跑”。

---

## 20. 当前 Agent 仍然有哪些局限

即便现在已经整理到比较完整的阶段，它仍然有局限：

- 本地知识库覆盖面仍然有限，不可能覆盖所有医学概念
- WHO 目前只接在术语解释上，没有变成通用医学知识层
- 长期记忆没有真正做摘要，只保留最近 N 轮短上下文
- 后台任务还是 Demo 级实现
- 医疗解释仍然属于健康信息科普，不是诊断系统

理解这些局限很重要，因为它能帮你分清：

- 这是一个结构完整的 Demo
- 但它还不是生产级医疗系统

---

## 21. 一句话总结这个 Agent

如果必须只用一句话概括它，我会这样说：

`这是一个先做安全和结构化准备，再让大模型负责综合表达的健康咨询 Agent。`

你如果能真正理解这句话在代码里的对应关系，这个 Demo 的核心你就已经吃透了。
