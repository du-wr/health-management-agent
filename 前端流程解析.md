# 前端流程解析

这份文档专门讲前端。

如果后端像工厂，前端就是用户真正接触到的“操作台”。它决定了这个 Demo 看起来是不是像一个真实产品，而不是一堆 API 的裸展示。

---

## 1. 前端技术栈

前端使用的是：

- React
- TypeScript
- Vite

同时，它还用到了浏览器原生能力：

- `fetch`
- `ReadableStream`
- `EventSource`

这意味着前端并没有引入特别重的状态管理或流式框架，而是用比较直接的方式完成：

- 文件上传
- SSE 订阅
- 流式问答显示

---

## 2. 前端代码从哪里开始

前端入口在：

- `frontend/src/main.tsx`

这里的作用很简单：

1. 挂载 React 根节点
2. 引入全局样式
3. 渲染 `App`

所以你真正需要重点看的文件其实是：

- `frontend/src/App.tsx`

---

## 3. `App.tsx` 是整个前端的主舞台

这个文件可以理解成前端的总控制台。

它负责维护几类状态：

- 报告状态
- 报告解析进度
- 对话消息
- 会话 ID
- 当前输入框内容
- 小结结果
- 错误提示
- 正在忙什么

你看到页面上所有主要功能，基本都从这个文件发起。

---

## 4. 这个页面现在有哪些区域

当前主页面主要有 3 块：

### 4.1 报告上传区

用户在这里上传 PDF / JPG / PNG。

### 4.2 健康咨询区

用户在这里输入问题，与 Agent 对话。

### 4.3 健康小结区

用户在这里生成和查看小结，并下载 PDF。

之前项目里还出现过单独的知识库展示区，但已经被去掉了，因为对最终演示价值不大。

---

## 5. 前端和后端通信都放在哪里

主要在：

- `frontend/src/api.ts`

这个文件相当于“前端的接口层”。

好处是：

- `App.tsx` 不需要直接写太多底层请求代码
- API 路径和 SSE 解析逻辑集中在一个地方

当前里面最重要的函数包括：

- `uploadReport`
- `streamChat`
- `streamReportProgress`
- `generateSummary`

---

## 6. 报告上传流程怎么走

这个流程从 `App.tsx` 的上传表单开始。

### 步骤 1：用户选文件

用户选择一个本地文件。

### 步骤 2：前端调用 `uploadReport`

`uploadReport(file)` 会把文件放到 `FormData` 里，再发送给：

- `/api/reports/upload`

### 步骤 3：前端收到初始报告对象

这时并不是解析完成，而只是知道：

- 报告 ID
- 文件名
- 当前状态

### 步骤 4：前端开始订阅报告进度

前端会调用：

- `streamReportProgress(reportId, onEvent)`

它内部使用的是：

- `EventSource`

### 步骤 5：界面实时更新

随着后端推送新的进度事件，前端不断刷新：

- 当前阶段名称
- 进度条
- 报告状态

### 步骤 6：收到最终报告结果

当后端推送 `final` 事件时，前端会把最终 `ReportParseResult` 放进状态里，后续问答和小结都可以围绕这份报告继续进行。

---

## 7. 为什么报告进度用 `EventSource`

因为报告解析是后台任务，不是一次请求里同步完成。

如果前端每 2 秒轮询一次，也能做到，但用户体验没那么好，而且代码不够优雅。

所以这里使用了 SSE。

你可以把它理解成：

“后端主动把状态变化推给前端，而不是前端不断追问后端。”

这也是为什么页面上能看到比较顺滑的解析进度变化。

---

## 8. 对话功能是怎么实现的

聊天部分有两个核心：

- 消息状态
- 流式事件处理

### 8.1 消息状态

在 `App.tsx` 中，每条消息通常包含：

- 唯一 ID
- 角色：用户或助手
- 文本内容
- 元数据
- 流式状态
- 状态标签

也就是说，聊天气泡并不只是纯字符串，而是一个带状态的对象。

### 8.2 发起提问

用户提交表单后，前端会：

1. 先把用户消息加到消息列表
2. 创建一个空的助手消息占位
3. 调 `streamChat(...)`

这样做的好处是：

- 用户能立刻看到自己的消息
- 助手区域也能立刻出现“正在生成”的占位状态

---

## 9. 流式聊天为什么看起来像打字机效果

因为 `streamChat(...)` 不是等完整答案后一次性返回，而是持续读取后端 SSE 流。

它内部用的是：

- `fetch`
- `response.body.getReader()`
- `TextDecoder`

然后前端会把后端发来的事件块解析成：

- `session`
- `status`
- `delta`
- `final`
- `error`

其中最关键的是：

### `delta`

表示一段新增文本。前端会把它追加到当前助手气泡中。

### `final`

表示最终完整结果。前端会用它收口：

- 写入最终 answer
- 保存 citations
- 保存 used_tools
- 保存 debug 信息

---

## 10. 为什么有“流式内容”和“最终结果”两个阶段

这是理解当前前端非常关键的一点。

在这个项目里，前端不是只接收一种数据，而是接收两层信息：

### 第一层：增量正文

给用户“正在生成”的感觉。

### 第二层：最终结构化结果

用于：

- 保存元数据
- 展示意图
- 展示工具
- 展示调试信息

这样做的好处是：

- 页面体验更像产品
- 同时又保留结构化信息，便于调试

---

## 11. 消息正文是怎么渲染的

在 `App.tsx` 里有一套轻量渲染逻辑：

- 标题识别
- 无序列表识别
- 有序列表识别
- `**加粗**` 识别
- 提示块识别

这不是完整 Markdown 渲染器，而是一个“够用且可控”的轻量解析器。

这么做的优点是：

- 实现简单
- 样式可控
- 不需要再引入额外的 Markdown 依赖

代价是：

- 支持的语法有限

---

## 12. 为什么前端能看到 `used_tools / debug`

因为后端的 `AgentResponse` 里带了这些元数据。

前端拿到后，会把它们放在助手消息的 `meta` 里。

于是每条消息下方都可以展示：

- 意图
- 工具
- 引用
- 调试信息

这对学习这个项目尤其有帮助，因为你可以直接观察：

- 这一轮到底被判成了什么问题
- 用了哪些工具
- 报告类问题的 `plan` 和 `synthesis` 长什么样

---

## 13. 调试信息为什么对理解 Agent 很重要

很多 Agent Demo 的问题在于“像黑盒”。

用户只看到答案，却不知道系统内部怎么想。

这个项目为了让学习过程更透明，把这些中间信息显式暴露出来：

- `analysis`
- `plan`
- `synthesis`
- `used_tools`

这让你在观察页面时，能把 Agent 的推理过程拆开来看。

对于小白来说，这一点非常有价值，因为你能真正看到：

- 输入分析做了什么
- 报告问题是怎么拆解的
- 综合异常是怎么整理的

---

## 14. 健康小结区怎么工作

小结区的逻辑相对简单。

用户点击生成后，前端会调用：

- `generateSummary({ session_id, report_id })`

然后拿到：

- Markdown
- PDF 路径
- 创建时间

页面会展示 Markdown 内容，并提供 PDF 下载链接。

这部分的思路是：

- 让报告最终能落成一个“像交付物”的结果

因为如果只是对话，不导出内容，产品感会弱很多。

---

## 15. `types.ts` 的作用是什么

前端类型都放在：

- `frontend/src/types.ts`

这个文件本质上是在告诉前端：

- 后端大概会返回什么数据结构

比如：

- `ReportParseResult`
- `AgentResponse`
- `SummaryArtifact`
- `ChatStreamEvent`

对于小白来说，可以把它当成“前后端契约说明书”。

你如果看不懂接口数据，先看 `types.ts` 往往比直接看业务逻辑更容易。

---

## 16. `styles.css` 在这个项目里重要吗

重要，但它的重要性不是业务逻辑层面的，而是体验层面的。

这个文件负责：

- 页面布局
- 聊天气泡样式
- 流式光标效果
- 状态胶囊
- 调试面板
- 进度区块

如果你只是想理解业务流程，可以后看它。

如果你想知道“为什么看起来像一个产品而不是接口测试页”，那就要看它。

---

## 17. 小白建议怎么读前端

建议顺序如下：

1. `frontend/src/main.tsx`
2. `frontend/src/App.tsx`
3. `frontend/src/api.ts`
4. `frontend/src/types.ts`
5. `frontend/src/styles.css`

先看：

- 页面做了什么

再看：

- 页面怎么请求后端

最后再看：

- 数据结构和样式

这样最不容易迷路。

---

## 18. 前端最值得学的地方是什么

如果你把这个 Demo 当作学习项目，前端最值得学习的不是复杂 UI，而是这 3 个点：

### 1. 用很少的代码做出完整产品流程

上传、解析、对话、小结都在一个页面串起来了。

### 2. 用 SSE 和流式响应改善体感

这让 Demo 看起来像一个真正的智能产品。

### 3. 把 Agent 的中间决策显式展示出来

这对调试和学习非常有帮助。

一句话总结前端：

`它不是为了做复杂页面，而是为了把 Agent 的流程尽可能清楚地展示给用户。`
